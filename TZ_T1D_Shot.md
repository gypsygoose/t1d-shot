# Техническое задание: T1D Shot

## 1. Общее описание проекта

**Название:** T1D Shot

**Назначение:** Мобильное приложение для людей с диабетом 1 типа, помогающее отслеживать места инъекций инсулина и обеспечивать максимальную ротацию мест уколов для профилактики липодистрофии и других осложнений.

**Платформа:** iOS / Android (кроссплатформенное)

**Технологический стек:**
- TypeScript
- React Native
- Expo (managed workflow, dev-режим через `expo start`)
- Локальное хранилище: **AsyncStorage** (`@react-native-async-storage/async-storage`) — см. обоснование ниже

**Хранение данных:** только локально на устройстве, без бэкенда и без облачной синхронизации. Данные должны надёжно сохраняться между запусками приложения, включая перерывы в несколько дней.

**Решение по хранилищу:** учитывая небольшой объём данных (ориентировочно 4–6 событий в день, то есть несколько десятков записей в месяц), рекомендуется использовать **AsyncStorage** (`@react-native-async-storage/async-storage`) — он проще в реализации, не требует схемы/миграций и полностью достаточен для такого объёма данных. Все события и состояния кнопок можно хранить как единый сериализованный JSON-объект (или несколько ключей), без необходимости в SQL-запросах. Переход на expo-sqlite в будущем возможен, если объём данных или требования к выборкам вырастут, но для MVP в этом нет необходимости.

   - Архитектура приложения и структура папок

   - Описание конечного автомата состояний кнопок (цвета/переходы)
   - Список зон и их кнопок с координатами/идентификаторами
   - Известные ограничения и TODO

5. Файл `CLAUDE.md` должен обновляться по мере разработки, чтобы Claude Code всегда имел актуальный контекст проекта.

## 3. Основной экран приложения

### 3.1 Визуализация тела
На экране отображается силуэт человека **только спереди** (вид сзади не требуется). Готовое изображение тела и общий дизайн интерфейса есть в Figma — макет нужно использовать как источник истины для визуального оформления, размещения зон и стилистики кнопок.

**Ссылка на Figma:** https://www.figma.com/design/grYg39698ogy0nEBd88Fup/Untitled?node-id=26-3

На странице представлены два макета:
- обычный режим (основной экран приложения);
- режим с открытой легендой (оверлей, см. п. 5.2).

Ассеты (изображение тела, иконки и т.п.) нужно экспортировать из Figma и добавить в проект.

На силуэте размещены **зоны введения инсулина** (например: живот слева/справа, бедра левое/правое, плечи левое/правое — точный список зон и их анатомическое положение нужно свести из Figma-макета в CLAUDE.md перед реализацией).

### 3.2 Кнопки мест инъекций
- В каждой зоне расположено несколько кнопок — конкретных точек инъекции.
- Каждая кнопка — это отдельный интерактивный элемент с собственным состоянием (цвет, дата последнего нажатия, заблокирован/активен).

## 4. Логика состояний и цветов кнопки

### 4.1 Базовый жизненный цикл цвета

| Состояние | Когда наступает |
|---|---|
| Белый (по умолчанию) | Кнопка ещё не использовалась, либо прошло 8+ дней с момента укола |
| Бордовый | Сразу после нажатия (день 0) |
| Красный | Через 1 день после укола |
| Тёмно-оранжевый | Через 2 дня |
| Оранжевый | Через 3 дня |
| Тёмно-жёлтый | Через 4 дня |
| Жёлтый | Через 5 дней |
| Тёмно-зелёный | Через 6 дней |
| Зелёный | Через 7 дней |
| Белый | Через 8 дней (возврат к исходному состоянию) |

### 4.2 Повторное нажатие (переиспользование места до полного восстановления)

Если пользователь нажимает на кнопку, которая **уже не белая** (то есть место недавно использовалось), кнопка переходит в состояние **чёрный (заблокировано системой)** на определённый срок, зависящий от того, в какой стадии цветового цикла она находилась в момент повторного нажатия:

| Состояние в момент повторного нажатия | Длительность чёрного состояния |
|---|---|
| Бордовый или красный | 4 дня |
| Тёмно-оранжевый или оранжевый | 2 дня |
| Тёмно-жёлтый или жёлтый | 1 день |

**Уточнено:** повторное нажатие на тёмно-зелёную или зелёную кнопку обрабатывается так же, как нажатие на белую кнопку — то есть кнопка просто становится **бордовой** (день 0) и запускает стандартный цикл с самого начала (п. 4.1), без перехода в чёрное состояние.

После истечения периода "чёрного" состояния кнопка возобновляет стандартный цикл, начиная с **красного**, и далее проходит по обычной цепочке (тёмно-оранжевый → оранжевый → ... → зелёный → белый), согласно таблице п. 4.1, но уже без бордовой стадии (так как чёрный сменяется сразу на красный).

### 4.3 Ручная блокировка (травма/синяк)

- Долгое нажатие (long press, ~2 секунды) на любую активную кнопку — независимо от её текущего цвета — переводит её в состояние **серый / неактивна**.
- В сером состоянии кнопка недоступна для нажатия (заблокирована для выбора как место укола).
- Повторное долгое нажатие (~2 секунды) на серую кнопку разблокирует её: кнопка возвращается к цвету, соответствующему её актуальному состоянию по времени, рассчитанному согласно правилам п. 4.1.
- **Уточнено:** отсчёт времени цветового цикла продолжается в фоне, пока кнопка находится в серой (ручной) блокировке. То есть сама блокировка не ставит цикл на паузу — она только скрывает/запрещает нажатие на кнопку внешне. После разблокировки кнопка сразу отображает тот цвет, который соответствует фактически прошедшему времени с момента последнего укола (как если бы блокировки не было).

### 4.4 Технические требования к конечному автомату

- Логику смены цветов рекомендуется реализовать как явный конечный автомат (state machine) с точным описанием состояний, переходов и таймеров.
- Все состояния и даты переходов должны пересчитываться на основе сохранённой даты последнего события (а не через постоянно работающие таймеры), чтобы корректно работать после закрытия/повторного открытия приложения.
- Пересчёт состояния кнопок должен происходить: при запуске приложения, при возврате в foreground, и опционально — по таймеру пока приложение открыто (например, раз в минуту), чтобы дневная смена цвета отражалась без перезапуска.

### 4.5 Индикатор последней нажатой кнопки в группе зон

Внутри кнопки, которая была нажата **последней в своей группе зон**, отображается галочка (checkmark), цвет которой подобран контрастно к цвету заливки кнопки, чтобы не сливаться с фоном (например, тёмная галочка на светлой заливке и наоборот).

Группы зон для этой индикации:
1. **Зоны бёдер** (левое и правое бедро вместе) — галочка отображается на кнопке, которая была нажата последней среди всех кнопок обеих зон бёдер.
2. **Зоны плеч и живота** (плечи и живот вместе) — галочка отображается на кнопке, которая была нажата последней среди всех кнопок этих зон.

Правила:
- В каждой группе в любой момент времени галочка может быть максимум на одной кнопке.
- При новом нажатии на кнопку внутри группы галочка переходит с предыдущей кнопки на только что нажатую.
- Если группа ещё не использовалась (нет ни одного события), галочка не отображается ни на одной кнопке группы.
- При использовании функции «Отменить» (п. 5.1) галочка должна пересчитываться и возвращаться на кнопку, которая нажата последней в группе после отмены (если такая есть).
- При «Очистить» (п. 5.3) все галочки сбрасываются вместе с остальными данными.

## 5. Нижнее меню

Меню из трёх кнопок в нижней части экрана:

### 5.1 «Отменить» (левая кнопка)
- Открывает попап-подтверждение.
- После подтверждения — отменяет последнее по времени событие (нажатие на кнопку зоны), возвращая соответствующую кнопку в предыдущее состояние (как если бы этого нажатия не было).
- **Уточнено:** «Отменить» действует **глобально** — отменяется самое последнее по времени событие среди всех кнопок/зон приложения, независимо от того, в какой зоне оно произошло.

### 5.2 «Легенда» (центральная кнопка)
- Открывает модальное окно/оверлей поверх всего интерфейса, кроме нижнего меню.
- Легенда описывает все зоны на силуэте (название зоны, её расположение).
- Также должна содержать расшифровку цветовой схемы (п. 4.1) для удобства пользователя.

### 5.3 «Очистить» (правая кнопка)
- Открывает попап-подтверждение.
- После подтверждения — полностью очищает все сохранённые данные о нажатиях на кнопки зон (сброс к исходному состоянию, все кнопки становятся белыми и разблокированными).

## 6. Модель данных (черновая, подлежит уточнению в CLAUDE.md)

Пример структуры события инъекции:

```ts
type ZoneGroup = 'thighs' | 'shoulders-and-belly'; // группировка для индикатора "последняя нажатая кнопка" (п. 4.5)

interface Zone {
  id: string;
  group: ZoneGroup;
  // название, положение и т.п. — свести из Figma
}

interface InjectionEvent {
  id: string;
  zoneId: string;
  buttonId: string;
  timestamp: number; // время нажатия
  type: 'injection' | 'blackout-reset'; // обычный укол или начало периода "чёрного" состояния
}

interface ButtonState {
  buttonId: string;
  zoneId: string;
  isManuallyBlocked: boolean; // серая блокировка вручную
  manualBlockedAt?: number;
  lastEvent?: InjectionEvent;
  isLastInGroup: boolean; // отображать ли галочку (вычисляемое поле, п. 4.5)
}
```

Все события должны храниться локально (AsyncStorage/SQLite — решение зафиксировать в CLAUDE.md) в виде истории, достаточной для работы функции «Отменить» и для корректного восстановления состояний после перезапуска приложения.

## 7. Нефункциональные требования

- Приложение должно корректно работать в offline-режиме (нет сетевых зависимостей).
- UI должен быть отзывчивым, анимации смены цвета кнопок — плавные, но не обязательные для MVP.
- Поддержка тёмной/светлой темы — опционально, уточнить приоритет.
- Локализация: минимум русский язык интерфейса (английский — опционально).
- Код должен быть покрыт базовым тестированием логики конечного автомата (расчёт цветов/состояний по датам) — это наиболее критичная для корректности часть приложения.
