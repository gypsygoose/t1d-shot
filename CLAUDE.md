# Insulin Shot Tracker — Project Documentation

## Overview

Mobile app for people with diabetes who take insulin injections to track insulin injection sites and enforce maximum rotation to prevent lipodystrophy. Users tap a point to log an injection; the point changes colour over a configurable number of days (1–8, default 8) to show when the site is safe to reuse. An optional second setting can additionally hard-disable re-marking a point for a configurable number of days after its last mark, on top of the colour cycle. A third, independent setting switches the app between this day-based Auto cycle and a Manual restore mode, where a point simply turns black the moment it's marked and stays disabled until the user clears it by hand.

**Platform:** iOS / Android — Expo Managed Workflow, TypeScript, React Native.

## Running the project

```bash
npm install
npm start          # opens Expo dev tools (scan QR with Expo Go)
npm test           # Jest unit tests (state machine logic)
npx tsc --noEmit   # TypeScript type check
```

**Every folder that exports something has an `index.ts`, and anything outside that folder imports through it — never by reaching into a concrete file inside it.** This applies uniformly: every component/hook/util folder (i.e. every `ComponentName/` entry with its own `hooks/`/`utils/`/`constants.ts`/`types.ts`) exports through its own `index.ts` (`export { ComponentName } from './ComponentName'`, plus any other re-exports a sibling file needs), so an import like `from "./MenuSheet"` keeps working unchanged whether the target used to be a single file or is now a folder — and the same rule extends to every other folder in the tree, including root-level `utils/`/`hooks/`/`data/`/`storage/`/`theme/`, every nested `utils/` folder (`logic/utils/`, `i18n/utils/`, `storage/utils/`, `theme/utils/`, and each component's own `utils/`), nested `hooks/` folders (`theme/hooks/`), `i18n/locales/`, and the umbrella `components/`, `components/common/`, `components/icons/`, and `screens/` folders. A barrel re-exports only what's actually consumed from outside the folder (`реекспортируй только то, что может пригодиться снаружи`) — a helper used only by a sibling file *inside* the same folder (e.g. `isValidEvent`/`isValidPointState`, used only by `isValidAppStorage.ts` within `storage/utils/`; `localDayIndex`, used only by `daysBetween.ts` within `logic/utils/`) is left out of that folder's `index.ts` entirely.

Two things do **not** change under this rule:

- **Same-folder sibling imports stay direct.** A file importing another file that sits right next to it in the same directory (e.g. `Dialog.tsx` importing `./Modal`, `Accordion.tsx` importing `./Chevron` — a subfolder of the same directory) never goes through that directory's own `index.ts`; it isn't "reaching into a folder from outside," and routing it through the barrel anyway would either be pointless (same folder) or, when the target is a subfolder whose contents are re-exported by an ancestor barrel that the importer itself is part of, genuinely circular.
- **A folder's barrel is never imported by code that folder's barrel (transitively) re-exports.** This is why `components/index.ts` (which transitively re-exports every component in the tree) is only ever imported from `screens/MainScreen/MainScreen.tsx` — the one consumer outside `components/` entirely — and never from, say, `BottomMenu.tsx` reaching for a sibling like `AutoLockDialog` (still `from "../AutoLockDialog"`, a direct file import, since `AutoLockDialog.tsx` has no folder/barrel of its own and both files live under the same `components/` barrel that would otherwise cycle). The same logic is why `i18n/LanguageContext.tsx` stays outside `i18n/index.ts`'s barrel (see the `i18n/` entry above) — a barrel can't re-export a file that itself depends on that barrel's own side effects.

## Coding conventions

- Use named exports for all components, functions, and modules — no `export default`. Import with `import { Foo } from "./Foo"`.
- The codebase is function/hook-based, not class-based — `StorageService` (`src/storage/StorageService.ts`) and `PointService` (`src/logic/PointService.ts`) are the exceptions: each groups a cohesive set of operations (AsyncStorage load/save/export/import; injection-site color computation/press handling) under a single static-methods-only class (`StorageService.loadStorage()`, `PointService.onPress()`, neither ever instantiated) instead of standalone exported functions, so callers get one qualified name (`StorageService.xxx`/`PointService.xxx`) at every call site instead of a long individual-function import list. A method reachable only from other methods on the same class (never from outside the class file) is marked `private` (e.g. `PointService.blackoutDurationFor`, called only by `PointService.onPress`) — one static method calling another on the same class still qualifies it with the class name (`PointService.computePointColor(...)`, not a bare call), matching how outside callers reach it.
- No single-letter or otherwise abbreviated identifiers for variables, parameters, or callback arguments (e.g. `zone` not `z`, `value` not `v`, `event`/`error` not `e`, depending on context) — always use the full, descriptive word, even in short arrow-function callbacks like `.map()`/`.filter()`.
- Every component (including small presentational helpers like icons or a single form field extracted from a screen) lives in its own file named after the component. Do not define a second component — even an unexported local one only used within the file — alongside another component in the same file. Icon components go in `src/components/icons/`. Generic, domain-agnostic UI primitives with no knowledge of app types (`Modal`, `Dialog`, `ConfirmDialog`, `ContextMenu`, `BottomSheet`, `Toast`, `TimeField`, `NumberPickerField`) go in `src/components/common/`; components that reference app domain types (e.g. `PointColor`, `ZoneId`, `AutoLockDialogMode`) or compose the app's screens stay directly under `src/components/`.
- **Helper function / hook placement.** Every standalone helper function lives in its own file in a `utils/` folder; every React hook lives in its own file in a `hooks/` folder — never define one inline alongside unrelated code. One used from more than one otherwise-unrelated place goes in the nearest shared root: app-wide, that's `src/utils/`/`src/hooks/`; scoped to one existing module folder (`storage/`, `logic/`, `i18n/`, `theme/`), that's `<module>/utils/`/`<module>/hooks/` (e.g. `src/storage/utils/isValidAppStorage.ts`, private to `storage.ts`; `src/theme/hooks/useTheme.ts`, the hook half of `ThemeProvider`/`useTheme()`, split out from `ThemeContext.tsx` alongside the created `ThemeContext` object it reads). `hooks/` itself doesn't get its own `utils/` subfolder this way — a non-hook helper shared across multiple files inside `hooks/` (e.g. `computeZoneBackfill`, used by three of `useAppStore.ts`'s own sub-hooks) still goes up to the shared `src/utils/` root, per the "never itself turned into a folder" rule below. One used by only a single component, or shared only across the handful of components built around it (e.g. `isSelectionEmpty`/`allMarks`/`allSettings`, used by `ExportOptionsDialog`/`ImportOptionsDialog`/`ClearOptionsDialog` but not app-wide), lives inside that component's own folder instead — `AppDataSelector/utils/` here, re-exported via `AppDataSelector/index.ts` for its sibling dialogs to import. **A hook file or a util file is never itself turned into a folder** — if a hook/util needs its own private constants, types, or further helpers, those go at the same directory level as the `hooks/`/`utils/` folder that contains it, not nested one level deeper (e.g. `useAppStore.ts`'s private `uuid`/`lastPressedByGroup`/`computeZoneBackfill` helpers live directly in the root `src/utils/`, alongside `useAppStore.ts`'s own `src/hooks/`, rather than in a `src/hooks/useAppStore/utils/` or a `src/hooks/utils/`).
- **A component without its own folder gets one (named after the component, e.g. `AppDataSelector/`, `SettingsSheet/`, `ZoneContainer/`) the moment it needs to place a private hook, util, constant, or type next to it — and loses that folder again (collapsing back to a single flat file) if it's ever simplified down to needing none, as `MenuSheet` was when its individual setting rows moved out into `SettingsSheet`.** Inside: `ComponentName.tsx` (the component itself — its `Props` interface and `StyleSheet` styles stay here, not extracted), a `hooks/`/`utils/` subfolder per kind of helper (each helper still its own flat file — the folder-per-component rule doesn't recurse into those), `constants.ts`, `types.ts`, and an `index.ts` that re-exports everything the rest of the app imports from it. A local constant or type that's private to that one file may stay declared inline instead of moving out, as long as there are 2 or fewer such declarations of that kind in the file; past that (more than 2 local constants, or more than 2 local types) — or the moment a single one of them is used outside that file — all of that kind move into `constants.ts`/`types.ts`. Helper functions have no such threshold: they always move to `utils/`, one per file, regardless of count or how private they are (see `isSelectionEmpty` in `src/components/AppDataSelector/utils/isSelectionEmpty.ts`).
- Types that enumerate string constants must be TypeScript `enum`s, not string-literal unions (e.g. `PointColor` in `src/types/point.ts`, `ZoneGroup`/`ZoneId` in `src/types/zone.ts`, `AppEventType` in `src/types/event.ts`, `AutoLockDialogMode` in `src/types/autoLock.ts`, `ToastStatus` in `src/types/toast.ts`, `ThemeMode` in `src/types/theme.ts`). Reference values as `EnumName.Member`, never as raw string literals.
- Discriminated-union result types (e.g. `PressResult` in `src/logic/types.ts`, `ImportResult` in `src/storage/types.ts`) use `type` as the discriminant field name (not `kind`), backed by its own enum (e.g. `PressResultType`, `ImportResultType`).
- No inline object type literals (`{ field: Type; ... }`) in a signature, prop, field, or `Record<K, V>` value position, even when only used once — always extract them into a named `interface`/`type` (e.g. `ZoneColorPair` in `src/theme/types.ts`, `MarkToastMessage` in `src/screens/MainScreen/utils/buildMarkToastMessage.ts`). A type used across multiple files goes in `src/types/`, split by entity into its own file (e.g. `Zone` in `src/types/zone.ts`, `PointAddress` in `src/types/point.ts`, `ExportedAppData` in `src/types/storage.ts`) and re-exported via `src/types/index.ts`'s barrel (`export * from './<entity>'`) so the rest of the app keeps importing from `../types`; one used only within a single file/component is still named, but declared locally in that file near its usage. A type scoped to one existing module folder rather than the whole app (e.g. `ThemeColors`/`ZoneColorPair`, used only inside `theme/` plus by `useTheme()` consumers) goes in that module's own `types.ts` (`src/theme/types.ts`) instead of `src/types/`.
- **Any function, method, or hook taking more than 2 parameters takes a single options object instead, destructured right in the signature**, backed by a named `interface` (never an inline object type literal, per the rule above — placed by the same rules: `src/types/` if used across files, a module's own `types.ts` if scoped to one folder, that component's own `types.ts`/local declaration if scoped to one file). Exactly 2 or fewer parameters stay as plain positional parameters — this only kicks in past 2. This applies even to `PointService.ts`'s required-argument methods (`computePointColor`, `daysUntilAvailable`, `onPress`, `colorLabel`) — collapsing their `daysToWhite`/`daysToAvailable`/`pointRestoreMode` (and other) parameters into one destructured object doesn't reintroduce the "silent default" risk described above, since a destructured field with no default in the signature is exactly as required by the compiler as a positional one; a call site omitting a field is still a compile error, not a silent fallback.
- No inline color literals (hex/`rgba`) in component styles. Chrome colors (surfaces, text, borders, icons, switches — anything that should flip between light/dark) are fields on `ThemeColors` (`src/theme/types.ts`), read via `const { colors } = useTheme()` (`src/theme/hooks/useTheme.ts`), never imported as flat constants. Colors that are deliberately theme-invariant (the `PointColor` cycle in `PointService.ts`, toast status colors) stay as plain constants in their own file/`src/constants.ts`, since they carry fixed semantic meaning independent of app chrome. A color literal used in 2+ places with the same semantic role is a shared constant (a `ThemeColors` field, or `src/constants.ts` if theme-invariant); one used in only one place is still a named constant, declared locally in the file/component where it applies (not inlined). Two literals that happen to share a value by coincidence, but mean different things (e.g. a UI accent color vs. an unrelated injection-cycle color in `PointService.ts`), are kept as separate constants — never merged just because the value matches. Exception: `ThemeColors.zoneColors` (`Record<ZoneType, ZoneColorPair>`, `src/theme/types.ts`) is a `ThemeColors` field but its two values (`DARK_COLORS`/`LIGHT_COLORS`) carry the *same* per-zone hue identity, not independent chrome — it's here only because the light theme needs a darker shade for legibility; see "Zones and points" below.
- The same rule applies to non-color literals: no unnamed "magic" numbers (durations, thresholds, sizes) or magic strings (storage keys, MIME types, format patterns) in component/logic code. A value reused in 2+ places for the same reason is a shared constant in `src/constants.ts` (or a shared helper in `src/utils/` for repeated formatting logic, e.g. `pad2.ts`, `splitSeconds.ts` — the latter also exporting `SECONDS_PER_MINUTE`); a single-use but deliberate value is still a named local constant in the file where it applies. One scoped to a single existing module folder rather than the whole app goes in that module's own `constants.ts` instead (e.g. `src/storage/constants.ts`'s AsyncStorage keys and `STORED_TRUE`/`STORED_FALSE`/`JSON_MIME_TYPE`/`EXPORT_FILE_PREFIX`/`IOS_JSON_UTI`, private to `storage.ts` and not re-exported by `storage/index.ts` since nothing outside `storage.ts` needs them). Ordinary one-off layout numbers (an arbitrary `padding`/`fontSize`/`borderRadius` with no cross-cutting meaning) don't need this — only values that encode an actual decision. Coincidental value matches with unrelated meaning are kept as separate constants, same as colors.
- **All user-facing text goes through i18next — never a hardcoded string literal.** Components call `useTranslation()`'s `t()` (or wrap rich inline-bold text in `<Trans i18nKey="..." components={{ bold: <Text style={boldStyle} /> }} />`, e.g. `HelpSheet.tsx`'s per-row explanations). Pure-logic files that don't render JSX (`PointService.ts`, `formatDateTime.ts`) stay free of an i18next dependency entirely: `PointService.colorLabel()` returns a `ColorLabelDescriptor` (`{ type: ColorLabelType; count?: number }`, the project's usual discriminated-union pattern) rather than a formatted string, and the one consumer (`HelpSheet.tsx`) formats it via `t()`; `formatDateTime.ts`'s `formatDateTime(timestamp, locale)` takes the active locale as an explicit parameter (`i18n.language` from the caller's `useTranslation()`) rather than reaching for a global. A `Record<SomeEnum, string>` of translation keys (e.g. `ZONE_LABEL_KEY` in `src/data/zones.ts`, `THEME_MODE_KEY`/`LANGUAGE_MODE_KEY` in `SettingsSheet/constants.ts`) is typed `Record<SomeEnum, TranslationKey>` (`TranslationKey` from `src/i18n/index.ts`, a dot-path union of every key in `locales/en.ts`) rather than plain `string`, so a value pulled from the map and handed to `t()` stays checked against the real key set. New text requires a key added to `src/i18n/locales/en.ts` and every other locale file (`ru.ts`, `es.ts`, `de.ts`, `fr.ts`, `tr.ts`, `pt.ts`) in the same change — `en.ts` is the source of truth (it exports the `AppLocale` type) and every other locale is typed as `AppLocale`, so a key missing from any of them is a compile error. Pluralized text always uses i18next's `count` option + key-suffix convention (`_one`/`_few`/`_many`/`_other` for Russian, `_one`/`_other` for English/Spanish/German/French/Turkish/Portuguese — see `common.daysCount` in `locales/en.ts`; the two-form locales still define `_few`/`_many` equal to `_other` purely to satisfy the shape shared with `ru.ts`, since their own plural rules never select those suffixes at runtime), never a hand-rolled plural helper.
- **Keep this file current**: whenever a change affects code style, project structure, or any other app-wide convention (not just a single file's internals), add or update the relevant note in this CLAUDE.md in the same change. Treat an out-of-date CLAUDE.md as a bug.
- **Keep README.md current too**: whenever a change affects the project description, features, setup/scripts, or structure, update README.md in the same change, not just CLAUDE.md.
- **Every new menu setting must round-trip through storage, export/import, and clear**, following the pattern set by `mirrored`/`autoLock*`/`daysToWhite`/`daysToAvailable`/`pointRestoreMode`/`gender`: its own AsyncStorage key + `load.../save...` static method pair on `StorageService` in `src/storage/storage.ts`, a field on `ExportedAppData` in `src/types/storage.ts`, a matching optional-type check in `isValidAppStorage`, inclusion in `useAppStore`'s `exportData`/`applyImport`/`clearSelected`, `BottomMenu`'s `buildImportData` filter, and a row label translation key documented in HelpSheet's "Пункты настроек"/`help.sectionSettingsItems` section (the "Пункты меню"/`help.sectionMenuItems` section right above it covers only the 4 top-level `MenuSheet` rows — Настройки/Экспорт/Импорт/Очистить — themselves). If the setting is a standalone toggle/value shown in `SettingsSheet` (not bundled into an existing row like auto-lock's three fields), also add it to `ExportSettingKey`, `AppDataSelector`'s `SETTING_LABEL_KEY` map, and `ImportOptionsDialog`'s `SETTING_AVAILABLE` map so it gets its own checkbox on the export, import, and clear sides alike — see "Selective export / merge import / clear" below. `themeMode`/`languageMode` follow the same storage/export/import/clear contract but, uniquely, aren't held in `useAppStore`'s state — see "Theme" and "Localization" below for why.

## Data model

```ts
// StoredPointState — persisted per point
interface StoredPointState {
  pointId: string;
  lastInjectionAt?: number;       // timestamp of last normal injection
  blackoutStartedAt?: number;     // timestamp when system blackout was triggered
  blackoutDurationDays?: number;  // duration of blackout in days
  isManuallyBlocked: boolean;     // gray (user-locked) state
}

// AppEvent — undo history entry. Capped at MAX_STORED_EVENTS (50,
// src/constants.ts): useAppStore's pressPoint/blockPoint/unblockPoint/
// markPointAt/clearPoint all append via the shared src/utils/appendEvent.ts
// helper, which drops the oldest event once the list would exceed the cap —
// Undo itself only ever pops the newest entry regardless of history length.
interface AppEvent {
  id: string;
  timestamp: number;
  type: 'injection' | 'blackout' | 'manual-block' | 'manual-unblock';
  pointId: string;
  zoneId: string;
  prevPointState: StoredPointState;  // snapshot for undo
}

// AppStorage — root object in AsyncStorage
interface AppStorage {
  pointStates: Record<string, StoredPointState>;
  events: AppEvent[];
}

// ExportedAppData — AppStorage fields plus settings that live in their own
// AsyncStorage keys, written out as one JSON file by export/import. Every
// field is optional: ExportOptionsDialog lets the user write only a subset
// of categories, and a missing field on import means "leave this category
// untouched", not "reset to default" — see "Selective export / merge
// import / clear" below. The same `ExportSettingKey`-keyed settings
// selection (plus a ZoneType-keyed marks selection) also drives
// ClearOptionsDialog, but there a *checked* category means "reset this to
// its default", not "write this to the file".
interface ExportedAppData {
  pointStates?: Record<string, StoredPointState>;
  events?: AppEvent[];
  mirrored?: boolean;
  autoLockEnabled?: boolean;
  autoLockAfterMarkSeconds?: number;
  autoLockAfterUnlockSeconds?: number;
  daysToWhite?: number;  // 1–8, default 8 — see "Point colour state machine" below
  daysToAvailable?: number; // 0–8, default 0, effectively capped at the current daysToWhite — see "Point colour state machine" below
  pointRestoreMode?: PointRestoreMode; // Auto/Manual, default Auto — see "Point restore mode" below
  themeMode?: ThemeMode; // Light/Dark/System, default System — see "Theme" below
  languageMode?: LanguageMode; // System/Russian/English/Spanish/German/French/Turkish/Portuguese, default System — see "Localization" below
  zonePointCounts?: ZonePointCounts; // per-ZoneType {rows, cols}, default = today's fixed counts — see "Zones and points" below
  enabledZones?: EnabledZones; // per-ZoneId on/off, default = every zone enabled — see "Zones and points" below
  gender?: Gender; // Male/Female, default Male — see "Gender" below
}
```

Storage keys: `@insulin_shot_tracker_v1` (pointStates/events), plus one key per setting (mirror, interface-locked, auto-lock, `daysToWhite`, `daysToAvailable`, `pointRestoreMode`, `gender`, `themeMode`, `languageMode`, `zonePointCounts`, `enabledZones`) — see `src/storage/constants.ts`, consumed by `StorageService` in `src/storage/StorageService.ts`.

## Selective export / merge import / clear

The **"Экспорт..."** row (`MenuSheet`) opens `ExportOptionsDialog.tsx` instead of exporting immediately; the **"Импорт..."** row similarly opens `ImportOptionsDialog.tsx` over the parsed file instead of applying it outright; the **"Очистить"** row similarly opens `ClearOptionsDialog.tsx` instead of clearing immediately. All three render the same `AppDataSelector` (`src/components/AppDataSelector/AppDataSelector.tsx`): a top-level **"Отметки точек укола"** row plus a **"Настройки приложения"** accordion of ten sub-checkboxes, one per `ExportSettingKey` (`Gender`/`Mirrored`/`AutoLock`/`EnabledZones`/`ZonePointCounts`/`PointRestoreMode`/`DaysToWhite`/`DaysToAvailable`/`Language`/`Theme`, `src/types/storage.ts`). The marks row is always keyed by `ZoneType` under the hood (`data/zones.ts`'s `ZONE_TYPES`/`ZONE_TYPE_LABEL_KEY`, reused as-is from "Zones and points" below — every locale already has `zones.groupShoulder`/`groupBelly`/`groupThigh` translated, so this needed no dedicated translation keys), but only `ClearOptionsDialog` passes `marksExpandable` `true`, rendering it as an accordion of three independent sub-checkboxes, **"Плечи"**/**"Живот"**/**"Бёдра"**, since a destructive clear scopes naturally to a body area; `ExportOptionsDialog`/`ImportOptionsDialog` leave `marksExpandable` at its default `false`, rendering the row as a single bulk checkbox that toggles all three zone types together — export and import always act on every mark at once, regardless of `StoredPointState.isManuallyBlocked`, or not at all; there's no way to export/import just one zone type. The settings accordion is built on the generic `Accordion` common component (`src/components/common/Accordion.tsx`), which knows nothing about checkboxes — `AppDataSelector` passes a `Checkbox` (with its own `label` prop) directly as its `label` node (the marks accordion, when rendered, follows the same pattern). In `Checkbox.tsx`, only the 22×22 box is a `TouchableOpacity`; the label text next to it is plain `Text`, not touchable. So tapping a group's bulk checkbox box bulk-toggles (and shows indeterminate for a partial selection of) its own sub-checkboxes — the marks accordion's rows (when shown) labeled via `ZONE_TYPE_LABEL_KEY`, the settings group's ten via `SETTING_LABEL_KEY` (same row-label translation keys as `SettingsSheet`) — while tapping the label text (or the chevron) falls through to that `Accordion`'s own header `TouchableOpacity` and toggles its own `expanded` instead, since React Native gives touch priority to the innermost responder and the text itself isn't one. Expanding/collapsing animates via `LayoutAnimation` (`Accordion`) and the chevron's rotation animates via `Animated.Value` (`Chevron`). The settings accordion (and, in `ClearOptionsDialog`, the marks accordion too) starts collapsed (independent `marksExpanded`/`settingsExpanded` state owned by each dialog), and neither selection is touched by expand/collapse. A checked row means something different in each of the three dialogs — "include in the export file", "apply from the import file", "reset to default" — but the UI and the `ExportSelection` shape are shared unchanged across all three (only `marksExpandable` differs).

`AppDataSelector` is a controlled component: it takes the current `ExportSelection` (`{ marks: Record<ZoneType, boolean>; settings: Record<ExportSettingKey, boolean> }`, `src/types/storage.ts`) and an `onSelectionChange`, an optional `marksExpandable` (default `false`) + `marksExpanded`/`onToggleMarksExpanded` (only read when `marksExpandable`) + `settingsExpanded`/`onToggleSettingsExpanded`, and two more optional props — `disabledMarksKeys` (a `ZoneType[]`) and `disabledSettingKeys` (an `ExportSettingKey[]`) — that grey out and disable individual rows; both default to "nothing disabled" (`[]`), which is exactly `ExportOptionsDialog`'s and `ClearOptionsDialog`'s case, since every category is always available to export or clear. It also exports `isSelectionEmpty(selection)` (checking every value of `selection.marks` directly via `Object.values`), used by all three dialogs to disable their Confirm button while nothing is selected, `SETTING_KEYS` (`Object.values(ExportSettingKey)`, `AppDataSelector/constants.ts`), and `allMarks(value)`/`allSettings(value)` (uniform-value `Record` builders, in `AppDataSelector/utils/` since `ExportOptionsDialog` and `ClearOptionsDialog` both build their own default selection from them — see the "Helper function / hook placement" coding-convention bullet above; `allMarks` iterates `data/zones.ts`'s `ZONE_TYPES` internally). `ExportOptionsDialog` owns the `ExportSelection` state (defaulting to everything checked, via `allMarks(true)`/`allSettings(true)`) and passes it straight through; its resulting selection is passed to `useAppStore`'s `exportData(themeMode, languageMode, dialogTitle, selection)` (`dialogTitle` is the share-sheet title, pre-formatted by the caller via `t('menu.exportOptionsDialog.shareDialogTitle')` since neither `useAppStore` nor `storage.ts` depend on i18next), which builds the `ExportedAppData` object with only the selected keys set — unselected categories are omitted entirely, not written with a default/empty value (see `ExportedAppData`'s comment above).

Unlike the settings group, where each `ExportSettingKey` gates its own independent scalar field, the marks group gates the whole `pointStates`/`events` pair at once: `exportData` writes both fields unfiltered the moment any `ZoneType` is checked (`Object.values(selection.marks).some(Boolean)`), and omits both entirely otherwise. Since `ExportOptionsDialog` always renders the marks row as a single bulk checkbox (`marksExpandable` left at its default `false`), every zone type ends up checked or unchecked together in practice, so this is really just "export marks or don't" — independent of `StoredPointState.isManuallyBlocked`, which no longer has any bearing on what gets exported.

Import mirrors this on the way back in: `isValidAppStorage` (used by `StorageService.pickImportFile`) only validates fields that are present, and `pickImportFile` returns the parsed data as-is (normalizing `pointStates` to include every current `POINTS` entry only when that category is present, rather than defaulting missing categories). `ImportOptionsDialog` derives `disabledMarksKeys`/`disabledSettingKeys` from whether each field is actually present on the parsed `ExportedAppData` (its `availableMarks`/`availableSettings` utils, e.g. `data.daysToWhite !== undefined`) and passes them into `AppDataSelector`, so a category absent from the file is forced unchecked and can't be selected — `availableMarks` marks every `ZoneType` available together whenever `pointStates`/`events` are both present, since a file's marks are never split by zone type on the export side; the marks/settings accordion checkboxes are each disabled by `AppDataSelector` once every one of their own sub-rows is (i.e. the file carries no marks, or no settings, at all) — and `Accordion`'s own `disabled` prop then also blocks expanding/collapsing that group. Confirming calls `BottomMenu`'s local `buildImportData(data, selection)`, which narrows the parsed file down to just the checked categories (mirroring `exportData`'s all-or-nothing marks gate on the way out), before handing that filtered object to `useAppStore`'s `applyImport`/`StorageService.importStorage`, which persist and merge only the fields present — an omitted category's current state and AsyncStorage value are left untouched rather than reset. This lets e.g. a settings-only export be imported without wiping injection history, or only auto-lock (not theme/language) be applied from a file that has both. `MainScreen`'s `onApplyImport` only calls `onSetThemeMode`/`onSetLanguageMode` when `data.themeMode`/`data.languageMode !== undefined`, for the same reason — which now also covers the user deliberately unchecking the theme/language row in `ImportOptionsDialog`.

`ClearOptionsDialog` is the third consumer, and the odd one out: unlike export/import, nothing is ever disabled (every category is always clearable, same as export), a checked category means "permanently reset this to its default" rather than "read/write a file" — and, uniquely, it passes `marksExpandable` `true`, so its marks group is choosable per `ZoneType` (Shoulder/Belly/Thigh) rather than all-or-nothing, since a destructive clear scopes more naturally to a body area ("clear just my belly points") than to manual-block status (`StoredPointState.isManuallyBlocked` has no bearing on what gets cleared either — a checked zone type resets every point in it, blocked or not). It owns its `ExportSelection` state the same way `ExportOptionsDialog` does — but with a different default from its "everything checked": since clearing is destructive and there's no single zone type that's a "safe" default, every mark and every setting starts unchecked (`allMarks(false)`/`allSettings(false)`) rather than reproducing the old all-or-nothing "Очистить" behavior by default; a user who wants that still can, just by checking every category first — and passes the confirmed selection to `useAppStore`'s `clearSelected(selection)`. For the marks group, `clearSelected` (`src/hooks/useClearSelected.ts`) filters `ZONE_TYPES` down to the checked ones, then resets every `pointStates` key whose own zone — recovered from its id prefix via `data/zones.ts`'s `zoneIdFromPointId`, not from the *current* `zoneData` (so an orphaned/out-of-range point id still lands in the right zone type) — maps (`ZONE_TYPE`) to one of the checked zone types, to a fresh `{ pointId, isManuallyBlocked: false }` (`src/utils/resetPointStates.ts`), and drops every `AppEvent` whose own `zoneId` maps to a checked zone type; an unchecked zone type's point states and events pass through unchanged, and the merged `pointStates`/`events` are persisted via `StorageService.saveStorage` immediately (this is the one persist call `clearSelected` makes eagerly, since it's the actual destructive action — unlike the settings resets below, there's no lazy-backfill safety net for a point history that's just been wiped). For the settings group, each checked `ExportSettingKey` is reset to the same default value `StorageService`'s own `loadXxx` falls back to when nothing is stored (`mirrored → false`, `autoLock → { enabled: false, afterMarkSeconds: DEFAULT_AUTO_LOCK_AFTER_MARK_SECONDS, afterUnlockSeconds: DEFAULT_AUTO_LOCK_AFTER_UNLOCK_SECONDS, deadline: null }`, `daysToWhite → DEFAULT_DAYS_TO_WHITE`), each persisted via that setting's own existing `StorageService.saveXxx`. `ZonePointCounts`/`EnabledZones` get the same combined-backfill treatment as `applyImport`'s equivalent branch: resetting either one to its default (`DEFAULT_ZONE_POINT_COUNTS`/`DEFAULT_ENABLED_ZONES`) can bring previously out-of-range/disabled slots back into range, so both are recomputed together against `buildZoneData` + `normalizeStorage` even when only one of the two is checked, so a clear that resets only one still backfills correctly against the other's current (not stale) value. `clearSelected` doesn't touch `themeMode`/`languageMode` directly — like `applyImport`, it leaves that to the caller: `BottomMenu` (which already holds `onSetThemeMode`/`onSetLanguageMode` as props for `ThemeDialog`/`LanguageDialog`) calls `onSetThemeMode(ThemeMode.System)`/`onSetLanguageMode(LanguageMode.System)` right after `onClear(selection)` whenever `selection.settings[ExportSettingKey.Theme]`/`[ExportSettingKey.Language]` is checked, rather than pushing that back up to `MainScreen` the way `onApplyImport` does — `BottomMenu` already owns every other piece of the clear flow (dialog visibility, the confirmation toast), so keeping the theme/language reset alongside it avoids splitting one user action across two components for no reason.

## Theme

The app supports light, dark, and system-matched themes via the **"Тема"** row in the menu (`ThemeDialog.tsx`, opened from `SettingsSheet`), backed by the `ThemeMode` enum (`Light`/`Dark`/`System`) in `src/types/theme.ts`. It's persisted with its own AsyncStorage key (`StorageService.loadThemeMode`/`.saveThemeMode` in `src/storage/storage.ts`) and round-trips through export/import as `ExportedAppData.themeMode`, defaulting to `ThemeMode.System`.

Unlike every other menu setting, `themeMode` is **not** held in `useAppStore`'s state — it's owned end to end by `ThemeProvider` (`src/theme/ThemeContext.tsx`), since `<ThemeProvider>` is mounted in `App.tsx`, above `MainScreen` (and therefore above the `useAppStore()` call that lives inside it). `src/theme/types.ts` defines the `ThemeColors` interface, and `src/theme/palette.ts` defines `DARK_COLORS`/`LIGHT_COLORS` — every chrome color (surfaces, text, borders, dividers, icons, switches, the bottom bar, dialogs/sheets/menus, the status bar). `ThemeProvider` loads the persisted `ThemeMode` on mount, resolves it against the live OS appearance (via React Native's `useColorScheme()`) into a `resolvedScheme` (`Light`/`Dark`) and the matching `ThemeColors`, and exposes `{ mode, resolvedScheme, colors, setMode }` through context — consumed anywhere in the tree, including `MainScreen` itself, with `const { colors } = useTheme();`. `MainScreen` reads `mode`/`setMode` (aliased to `themeMode`/`onSetThemeMode`) to pass down to `BottomMenu`, and threads `themeMode` into `useAppStore().exportData(themeMode, ...)` and calls `setMode(data.themeMode)` after `applyImport(data)`, since `useAppStore` itself no longer tracks or persists this setting.

Domain/status colors are deliberately **not** theme-dependent and stay as plain constants: the injection-cycle `PointColor` palette (`COLOR_HEX` in `src/logic/constants.ts`) and the toast status colors (`TOAST_*_COLOR`/`TOAST_*_BACKGROUND_COLOR`/`TOAST_TEXT_COLOR` in `constants.ts`) carry fixed semantic meaning (day count, message severity) and are self-contained saturated colors that read fine on either background. `BodyImage` (`src/components/BodyImage/`) and both of the variants it switches between (`MaleBodyImage`/`FemaleBodyImage`) are likewise reused unchanged in both themes (they're solid-filled illustrations, not linework that depends on the surface behind it). One exception to "domain colors are theme-invariant": the per-zone `accent`/`glow` pair (`ThemeColors.zoneColors`, keyed by `ZoneType`) — `ZoneContainer`'s zone block, `InjectionPoint`'s glow halo, and `HelpSheet`'s zone-legend badge/text all read too faint against the light theme's bright background at `DARK_COLORS.zoneColors`' values, so `LIGHT_COLORS.zoneColors` holds a darker shade of the same per-zone hue instead — see "Zones and points" below. The zone's hue identity is unchanged; only its rendering against theme-dependent chrome gets theme-selected.

## Localization

All user-facing text is localized into Russian, English, Spanish, German, French, Turkish, and Brazilian Portuguese via **i18next**/**react-i18next** (`src/i18n/`) — see the "All user-facing text goes through i18next" coding-convention bullet above for the per-file authoring rules. This section covers the runtime architecture.

`src/i18n/locales/en.ts` is a single nested object (`common`/`zones`/`toast`/`menu`/`pointMenu`/`markDialog`/`mainScreen`/`help`/`stateMachine`, grouped to mirror the component structure) and is the source of truth for the translation key shape — it exports `AppLocale` (`typeof en`); `ru.ts`/`es.ts`/`de.ts`/`fr.ts`/`tr.ts`/`pt.ts` each import that type and declare `export const xx: AppLocale = {...}`, so TypeScript rejects any of them if a key is missing or misshapen. `src/i18n/index.ts` combines all seven into `resources`, derives and exports `TranslationKey` (a dot-path string-literal union of every leaf key in `en`, via a recursive mapped type — used to type `Record<SomeEnum, TranslationKey>` lookup maps like `ZONE_LABEL_KEY`), and calls `i18next.init()` once at module load with a best-guess initial language read synchronously from `expo-localization`'s `Localization.getLocales()` (avoids a flash of the wrong language before the persisted override loads). `src/i18n/i18next.d.ts` augments `i18next`'s `CustomTypeOptions` so every `t('...')` call app-wide — not just ones going through a lookup map — is checked against `en`'s key shape.

The **"Язык"** row in the menu (`LanguageDialog.tsx`, opened from `SettingsSheet`) offers System/English/Russian/German/Spanish/French/Turkish/Portuguese, backed by the `LanguageMode` enum (`System`/`Russian` = `'ru'`/`English` = `'en'`/`Spanish` = `'es'`/`German` = `'de'`/`French` = `'fr'`/`Turkish` = `'tr'`/`Portuguese` = `'pt'`, the seven concrete values doubling as real i18next language codes) in `src/types/language.ts`. It follows the exact same architecture as `ThemeMode`/`ThemeProvider` above: **not** held in `useAppStore`'s state, instead owned end to end by `LanguageProvider` (`src/i18n/LanguageContext.tsx`, mounted in `App.tsx` alongside `ThemeProvider`), with its own AsyncStorage key (`StorageService.loadLanguageMode`/`.saveLanguageMode` in `src/storage/storage.ts`) and export/import round-trip as `ExportedAppData.languageMode`. `LanguageProvider` loads the persisted mode on mount, resolves `System` against the device locale (via `getDeviceLanguage()` in `src/i18n/utils/deviceLanguage.ts`, read once per mode change rather than subscribed to live OS locale changes — unlike light/dark, a mid-session locale change is practically never observed) into a `resolvedLanguage` (`Russian`/`English`/`Spanish`/`German`/`French`/`Turkish`/`Portuguese`), and calls `i18next.changeLanguage()` whenever it changes. `getDeviceLanguage()` only looks at `Localization.getLocales()[0]` (the device's top-priority preferred language), not the whole list — devices/simulators commonly report English as a secondary fallback entry even when the primary system language is Russian, so checking the full list with `.some(...)` would misresolve `System` to English on exactly those devices; it maps that top-priority `languageCode` against a small `DEVICE_LANGUAGE_CODE_MODE` lookup (`ru`/`es`/`de`/`fr`/`tr`/`pt`), falling back to English for any other device language; `pt` is used as-is regardless of region (`pt-BR` vs `pt-PT` both resolve to the same Brazilian-Portuguese translation, since `Localization.getLocales()[0].languageCode` only reports the ISO 639-1 language, not the region — the app has no separate European-Portuguese locale); `index.ts`'s initial best-guess (before the persisted mode loads) shares this same helper so the two can't drift out of sync — this is what actually drives every `useTranslation()` consumer's re-render, not the `LanguageMode` value itself. `MainScreen` reads `mode`/`setMode` (aliased `languageMode`/`onSetLanguageMode`) from `useLanguage()` the same way it reads `useTheme()`, threads `languageMode` into `exportData`, and calls `setMode(data.languageMode)` after `applyImport(data)`.

Two pure-logic files stay deliberately free of an i18next dependency despite producing locale-sensitive output, per the coding-convention bullet above: `PointService.colorLabel()` returns a `ColorLabelDescriptor` that `HelpSheet.tsx` formats via `t()`, and `formatDateTime.ts`'s `formatDateTime(timestamp, locale)` takes the locale explicitly (backed by `Intl.DateTimeFormat`, so `DD.MM.YYYY` for `ru` vs `MM/DD/YYYY` for `en` falls out automatically) instead of reaching for the i18next singleton itself.

`APP_NAME` (`src/constants.ts`) is the one piece of user-visible text deliberately **not** localized — it's a brand name, not prose.

## Gender

A fourth independent setting — **"Пол"** (the first row in `SettingsSheet`, opening `GenderDialog` — a `common/RadioGroup` over the two options, same pattern as `PointRestoreModeDialog`/`ThemeDialog`/`LanguageDialog`) — controls which body silhouette renders behind the injection points, backed by the `Gender` enum (`Male`/`Female`, `src/types/gender.ts`), default `Male` (`DEFAULT_GENDER` in `src/constants.ts`) so existing installs — which only ever had the one silhouette — see no visual change. Like `pointRestoreMode`/`daysToWhite`, it's held directly in `useAppStore`'s state (not in its own top-level Provider like `themeMode`/`languageMode`), since it has no reason to live above `MainScreen`. It's persisted like `pointRestoreMode` (its own AsyncStorage key, round-trips through export/import as `ExportedAppData.gender`/`ExportSettingKey.Gender`) — see "Selective export / merge import / clear" above.

Unlike `pointRestoreMode`, `gender` has no effect whatsoever on point coloring, blackout/availability logic, or any other app behavior — it purely picks which SVG `BodyImage` (`src/components/BodyImage/`) renders as the background layer behind `MainScreen`'s `ZoneContainer`s. `BodyImage.tsx` takes a `gender: Gender` prop (plus the same optional `style` its old `BodyIllustration` predecessor took) and renders either `MaleBodyImage` or `FemaleBodyImage` — both private to the `BodyImage/` folder, each an inline `react-native-svg` tracing of the Figma male/female illustrations with its own native `viewBox` (627×1081 male, 608×1073 female). Zone/point positions (`data/zones.ts`) are unaffected by which variant is shown — they're tuned as fractions of the body image *container*, not of either illustration's own intrinsic proportions, and the caller's `style` (`MainScreen`'s `imageContainer`/`image`, both `width`/`height: "100%"`) governs the actual rendered box for either variant identically.

## Point colour state machine

### Days-to-white setting

The number of days it takes an injection point to reach White (fully free) is configurable via the **"Дней до восстановления точки"** row in the menu (`DaysToWhiteDialog.tsx`), range 1–8, default 8 (`MIN_DAYS_TO_WHITE`/`MAX_DAYS_TO_WHITE`/`DEFAULT_DAYS_TO_WHITE` in `src/constants.ts`). It's persisted like `mirrored`/auto-lock (its own AsyncStorage key, round-trips through export/import as `ExportedAppData.daysToWhite`) and threaded into `PointService.computePointColor`/`PointService.onPress` as a required third argument — every call site must pass the caller's actual current `daysToWhite`, never a stand-in default (see the `PointService.ts` folder-structure entry above).

At the default of 8, every color below is used, one per day. Lowering the setting compresses the cycle by dropping colors, in this fixed order (first dropped → last dropped): **dark yellow → dark orange → dark green → orange → red → green → yellow**. Maroon (day 0) is never dropped; White is always reached on day `daysToWhite`. See `PointService.activeCycleColors` in `src/logic/PointService.ts` / `COLOR_REMOVAL_ORDER` in `src/logic/constants.ts`. Examples:

- 7: orange on day 3, yellow on day 4, green on day 6, white on day 7 (dark yellow dropped)
- 6: red on day 1, orange on day 2, yellow on day 3, green on day 5, white on day 6 (dark yellow, dark orange dropped)
- 1: white on day 1 (only maroon remains)

Blackout durations (below) stay keyed by color identity regardless of the setting — a color's blackout duration doesn't change just because other colors were dropped from the cycle.

### Days-to-available setting

A second, independent setting — **"Дней до доступности точки"** (`DaysToAvailableDialog.tsx`) — controls how many days after the start of a point's current cycle (the last injection, or the moment a system blackout ends) marking it again stays disabled *on top of* the color cycle above. Range 0–`daysToWhite`, default 0 (`MIN_DAYS_TO_AVAILABLE`/`DEFAULT_DAYS_TO_AVAILABLE` in `src/constants.ts` — there's no fixed `MAX_DAYS_TO_AVAILABLE` constant; the dialog's own `maxDays` prop is always the *current* `daysToWhite`). It's persisted like `daysToWhite` (its own AsyncStorage key, round-trips through export/import as `ExportedAppData.daysToAvailable`/`ExportSettingKey.DaysToAvailable`) and threaded into `PointService.onPress`/`PointService.daysUntilAvailable` as a required fourth argument — every call site must pass the caller's actual current `daysToAvailable`, never a stand-in default (see the `PointService.ts` folder-structure entry above; `MainScreen/utils/buildMarkToastMessage.ts`'s re-run of `onPress` to detect a `Blackout` outcome for the mark-confirmation toast takes and passes the real value too, for the same reason — see "Point address" below).

The setting's *effective* cap is always `Math.min(daysToAvailable, daysToWhite)`, computed fresh at every read in `PointService.daysUntilAvailable` — the raw stored value is never rewritten just because `daysToWhite` changed. So setting `daysToAvailable` to 5 while `daysToWhite` is 8, then lowering `daysToWhite` to 3, makes the point available in 3 days (not 5); raising `daysToWhite` back to 8 later restores the original 5-day wait, since the raw setting was never touched. `daysUntilAvailable(state, now, daysToWhite, daysToAvailable)` returns the number of days still remaining, or `undefined` once the point can be marked — White is **always** immediately available regardless of the setting (an availability window can never exceed reaching White itself, since the effective cap never exceeds `daysToWhite`); Gray/Black are unaffected too, since they're gated by their own, unrelated priority rules (see below) — a point stuck in an active cycle color (Maroon through Green, dark-green/green included) is unavailable until `daysSinceCycleStart >= effective cap`. `daysSinceCycleStart` (`src/logic/utils/daysSinceCycleStart.ts`) is the same day count `computePointColor` derives internally, just exposed standalone so `daysUntilAvailable` doesn't need to reimplement the blackout/injection branching.

`PointService.onPress` checks availability right after the existing Gray/Black check and before the White/dark-green/green vs. blackout branch, returning a new `PressResultType.Unavailable` (`{ type: PressResultType.Unavailable; daysRemaining: number }` in `src/logic/types.ts`) when gated. `useAppStore`'s `pressPoint`/`markPointAt` treat `Unavailable` as a no-op, same as `Blocked`. `MainScreen.handlePress` pre-checks `PointService.daysUntilAvailable` before calling `actions.pressPoint` (mirroring its existing Gray/Black pre-check) and shows a `toast.pointUnavailable` toast (pluralized, `ToastStatus.Info`) instead of marking — the point's color is completely unaffected by this (per its own comment: "if a point isn't available for marking, its color stays exactly what it would be at daysToAvailable = 0"), and `ZoneContainer`/`InjectionPoint`'s `unavailable` prop instead fills the knob's top-right half with a gray overlay (`UNAVAILABLE_OVERLAY_COLOR` in `InjectionPoint/constants.ts`) so the underlying color cycle stays fully visible. `PointContextMenu` mirrors this: its `daysUntilAvailable` prop (computed by `MainScreen` the same way as `menuPointColor`) disables the "Отметить" `ContextMenuItem` and adds a `pointMenu.availableIn` info line (pluralized) stating how many days remain — same information the tap-toast and the disabled menu item convey, just in the two different places a user might try to mark an unavailable point. `HelpSheet`'s "Color scheme" legend (`help.sectionColorScheme`) has one further, non-interactive example row demonstrating the overlay itself (`help.colorScheme.unavailableExample`), appended after `colorOrder`'s per-`PointColor` swatches.

### Point restore mode

A third, independent setting — **"Режим восстановления точек"** (a row in `SettingsSheet`, directly below auto-lock, opening `PointRestoreModeDialog` — a `common/RadioGroup` over the two options, same pattern as `GenderDialog`/`ThemeDialog`/`LanguageDialog`) — switches the whole app between two mutually exclusive coloring/press-handling strategies, backed by the `PointRestoreMode` enum (`Auto`/`Manual`, `src/types/pointRestoreMode.ts`), default `Auto` (`DEFAULT_POINT_RESTORE_MODE` in `src/constants.ts`). Unlike `themeMode`/`languageMode`, it's held directly in `useAppStore`'s state (like `daysToWhite`/`daysToAvailable`), since it has no reason to live above `MainScreen`. It's persisted like `daysToWhite` (its own AsyncStorage key, round-trips through export/import as `ExportedAppData.pointRestoreMode`/`ExportSettingKey.PointRestoreMode`) — see "Selective export / merge import / clear" above.

`Auto` is every behavior described elsewhere in this section, unchanged. `Manual` replaces it entirely: `PointService.computePointColor` short-circuits right after the existing Gray check (manual block still takes priority over everything, in both modes) and returns `PointColor.White` if the point has never been marked (`lastInjectionAt === undefined`) or `PointColor.Marked` otherwise — the day-based cycle, system blackout, and the days-to-available countdown never apply. `PointColor.Marked` (`src/types/point.ts`) is a color in its own right, deliberately distinct from `PointColor.Black` (system blackout, only ever produced in `Auto`) even though both share the same near-black hex (`COLOR_HEX[PointColor.Marked]`, `src/logic/constants.ts` — a coincidental value match kept as a separate constant, per the coding-convention bullet on color literals above) — the two must never be conflated, since `Marked` never expires on its own the way a system blackout does. `PointService.daysUntilAvailable` returns `undefined` unconditionally in `Manual` (there's no partial-day gating to report — a point is either available or blocked outright, and that's already conveyed by its color), and `PointService.onPress` blocks (`PressResultType.Blocked`) on Gray/Black/Marked alike, otherwise marks the point (`PressResultType.Injection`) with no blackout/availability branching. Practically: a point starts White, turns permanently `Marked` (black) the moment it's tapped, and stays that way — tapping it again, or trying to mark it via the context menu's "Отметить" item (which `PointContextMenu` disables whenever `color === PointColor.Marked`, the same way it already disables that item for the days-to-available countdown) — is a no-op until the user clears the point (`common.clear`, in the context menu or `ClearOptionsDialog`), which resets it back to a fresh, White `StoredPointState` exactly like it does today.

Selecting `Manual` disables (dims + no-ops, doesn't hide) the "Дней до восстановления точки"/"Дней до доступности точки" rows in `SettingsSheet`, since neither setting has any effect while `Manual` is active — their stored values are left completely untouched, so switching back to `Auto` restores whatever `daysToWhite`/`daysToAvailable` were configured before. `HelpSheet`'s color-scheme legend (`help.sectionColorScheme`) reduces to exactly three rows in `Manual` — White (free), Marked (black, `stateMachine.colorLabel.marked`), Gray (manually blocked) — via `colorOrder(daysToWhite, pointRestoreMode)` (`HelpSheet/utils/colorOrder.ts`) branching before building the normal Auto list, and the unavailable-overlay example row right after it is skipped entirely (`Manual` never produces that overlay, since `daysUntilAvailable` is always `undefined`). White's own label text also differs by mode: `PointService.colorLabel` takes `pointRestoreMode` as a required third argument (same no-default-values rule as `computePointColor`/`onPress`) and, for `PointColor.White` in `Manual`, returns a `ColorLabelType.WhiteManual` descriptor (no `count`) instead of the usual `ColorLabelType.White`/`count: daysToWhite` — `Manual`'s White has no day-based cycle to report a count against, so `formatColorLabel.ts` formats it via a separate, non-pluralized `stateMachine.colorLabel.whiteManual` key ("Свободно"/"Free"/...) rather than reusing `white_one`/`white_few`/`white_many`/`white_other`'s "unused for N+ days" copy.

### Normal injection cycle (days since `lastInjectionAt`)

"Days" are local calendar days, not elapsed 24h periods: the color advances at local midnight (device timezone), so a press at 15:30 becomes day 1 as soon as the clock crosses into the next calendar day, not 24h later. Same rule applies to the post-blackout cycle. See `daysBetween`/`localDayIndex` in `src/logic/utils/`.

Full (default, daysToWhite = 8) cycle:

| Days | Colour |
|---|---|
| 0 | Maroon `#7B1D1D` |
| 1 | Red `#DC2626` |
| 2 | Dark orange `#C2410C` |
| 3 | Orange `#EA580C` |
| 4 | Dark yellow `#A16207` |
| 5 | Yellow `#EAB308` |
| 6 | Dark green `#166534` |
| 7 | Green `#16A34A` |
| 8+ | White `#EBEBEB` |

### Re-use (blackout) — point pressed while not white/dark-green/green

| Colour at press | Blackout duration |
|---|---|
| Maroon / Red | 4 days |
| Dark orange / Orange | 2 days |
| Dark yellow / Yellow | 1 day |

After blackout: post-blackout cycle starts at **Red** (maroon is skipped), counting from `blackoutStartedAt + blackoutDurationDays * DAY_MS`. The post-blackout cycle is subject to the same daysToWhite-driven compression as the normal cycle (with Maroon excluded, since it never appears post-blackout).

Dark-green / Green re-press → treated as white (restart cycle from maroon, no blackout).

### Manual block (gray)
Long-press (~800 ms) → toggle `isManuallyBlocked`. Gray overrides all colours visually, but the underlying time cycle continues ticking.

### Priority (highest → lowest)
1. Gray (manual block)
2. Black (system blackout, if `blackoutStartedAt > lastInjectionAt` and `now < blackoutEnd`)
3. Post-blackout cycle (if blackout ended)
4. Normal injection cycle
5. White (never used or 8+ days)

This priority list is purely about *color*, and applies only when point restore mode is `Auto` — the days-to-available setting above is a separate, orthogonal check applied only at press time (`PointService.onPress`/`daysUntilAvailable`), not a 6th color tier; it never changes what color a point renders as. In `Manual` restore mode, this whole priority list is replaced by the much shorter Gray → Marked → White check described in "Point restore mode" above.

## Toast statuses

Every toast (`src/components/common/Toast.tsx`) carries a `ToastStatus` (`src/types/toast.ts`) that drives its background/border color and leading icon (`src/components/icons/Toast{Info,Warn,Success,Error}Icon.tsx`, colors in `src/constants.ts`):

| Status | Used for |
|---|---|
| `Info` | Locking/unlocking the interface (bottom-bar lock button); auto-lock engaging on its own (countdown elapsed); attempting to mark a point while the interface is locked, a point that's individually blocked (gray/black), or a point still gated by the days-to-available setting (see "Point colour state machine" above) |
| `Warn` | A point mark that triggers a system blackout (site reused too early) |
| `Success` | A point mark with no blackout; manual block/unblock of a point; clearing a point; clearing selected data; undo; successful export/import; any menu setting change (mirror toggle, auto-lock enable/disable/edit, point restore mode, gender, days-to-white, days-to-available) |
| `Error` | A failed import (corrupt/invalid file) |

`MainScreen`'s `showToast` (message, status, optional duration — defaults to `TOAST_DURATION_MS`) covers point-level actions (mark/block/clear, both driven off the shared `buildPointAddressSuffix` helper) and the interface lock toggle. `BottomMenu` receives it as an `onNotify` prop for actions it owns end-to-end (undo, clear, export, import success/failure, and every `MenuSheet`/`SettingsSheet`/`AutoLockDialog`/`DaysToWhiteDialog` setting change), since those confirm/cancel flows live entirely inside that component. `useAppStore` itself takes an optional `onAutoLockFired` callback (read through a ref so passing a fresh closure each render doesn't retrigger the countdown effect), invoked only when the auto-lock countdown elapses on its own — not when the user locks the interface manually, which the caller already notifies itself.

Toasts stack: each new one renders above older ones (`ToastStack`, `src/components/common/ToastStack.tsx`, rendering a `ToastEntry[]` newest-first) instead of replacing them, and each entry (`Toast`, `src/components/common/Toast.tsx`) fades itself in/out and self-dismisses after its own `duration`. `MainScreen` caps the stack at `MAX_STACKED_TOASTS` (`src/constants.ts`), dropping the oldest entry once a new one would exceed it.

**Every completed user action must show a toast.** When adding a new one (a new point action, a new menu setting, anything that completes an operation), don't pick its message/status unilaterally — propose the toast text and the suggested `ToastStatus` and let the user confirm that status, pick a different one, or decline the toast entirely before wiring it in.

## Zones and points

6 zones, 30 points by default. All point positions are fractions (0–1) of the body image container (393.46×621.91), extracted directly from the Figma "with buttons" frame (node `27:744`, file `grYg39698ogy0nEBd88Fup`). This container aspect ratio is independent of the illustration actually rendered inside it — `BodyImage` (see `components/`' folder structure above) is stretched to fill it exactly, regardless of the illustration's own native proportions. Each point renders as a 25 px knob (state colour fill) inside a 35 px radial-gradient glow halo tinted with its zone's `glow` colour (see `ThemeColors.zoneColors` in `src/theme/types.ts`).

### Configurable point grid

The number of rows and columns is configurable per zone **type** (`Shoulder`/`Belly`/`Thigh` — shared by both zones of a left/right pair, not per individual `ZoneId`) via the **"Точки в зонах"** row in the menu (`ZonePointsDialog`, opened from `SettingsSheet`), minimum 1×1 for every type (`MIN_ZONE_ROWS`/`MIN_ZONE_COLS` in `src/constants.ts`) but a per-type maximum (`ZONE_MAX_GRID` in `src/data/zones.ts`: shoulders 3 rows × 2 cols, belly 4×4, thighs 4×3 — not every zone has the same amount of usable space), backed by `ZonePointCounts` (`Record<ZoneType, ZoneGridConfig>`, `src/types/zone.ts`) and defaulting to today's fixed counts (`DEFAULT_ZONE_POINT_COUNTS` in `src/data/zones.ts`: shoulders 3×1, belly 3×3, thighs 3×2). `ZonePointsDialog` builds each type's own row/column `NumberPickerField` options from its own `ZONE_MAX_GRID` entry (`buildRangeOptions` in its `utils/`). It's persisted like `daysToWhite` (its own AsyncStorage key, round-trips through export/import as `ExportedAppData.zonePointCounts`/`ExportSettingKey.ZonePointCounts`) — see "Selective export / merge import / clear" above.

`SettingsSheet`'s own "Точки в зонах" row shows the current total point count next to the label as a plain number, no i18next key (`SettingsSheet/utils/countActivePoints.ts` sums `rows * cols` over every *enabled* `ZoneId`, so a disabled zone's grid doesn't inflate the total) and, below the row, one line per `ZoneType` with at least one enabled side (`SettingsSheet/utils/buildZonePointsSummary.ts`), e.g. "Плечи: 2x1" — a plain `${rows}x${cols}` figure (not pluralized prose: an earlier version spelled it out as "2 ряда, 5 колонок", built from two independently-pluralized sub-strings interpolated into a locale-specific template, but a bare grid dimension reads the same in every locale and the pluralization overhead across 7 locales wasn't worth it) interpolated into the existing `toast.labeledValue` template ("{{label}}: {{value}}") the same way `buildActiveZonesSummary.ts` does. A `ZoneType` with neither side enabled is skipped entirely (it renders no points regardless of its configured grid); if every zone is disabled, the summary collapses to the single `menu.allZonesDisabledSummary` sentence instead (shared with `buildActiveZonesSummary.ts` below — see "Enabling/disabling individual zones").

Because the grid size is now dynamic, `ZONE_LAYOUT`/`POINTS`/`POINT_MAP`/`POINTS_BY_ZONE`/`POINT_ADDRESS` are no longer static module-level consts: `data/zones.ts`'s `buildZoneData(zonePointCounts, enabledZones)` computes all of them fresh (bundled as `ZoneRuntimeData`, `src/types/zone.ts`) from the current settings (`enabledZones` — see "Enabling/disabling individual zones" below). `src/hooks/useZoneSettings.ts` memoizes this as `zoneData` (`useMemo` keyed on the `zonePointCounts`/`enabledZones` it's passed) and `useAppStore` returns it alongside the rest of its state; `MainScreen` threads `zoneData.zoneLayout`/`zoneData.pointsByZone` into each `ZoneContainer` as props (no longer static imports there) and `zoneData.pointMap`/`zoneData.pointAddress` into `PointContextMenu`'s `address` prop and the `buildPointAddressSuffix`/`buildMarkToastMessage` helpers. Internally, `useZoneSettings` also owns the `zoneDataRef` that `usePointActions`'s `pressPoint`/`blockPoint`/`unblockPoint`/`markPointAt`/`clearPoint` and `useImportExport`'s `pickImportFile` read the live value through (same ref-through-render pattern `useAutoLock`'s `onAutoLockFiredRef` uses), so those `useCallback`s can stay stable without needing `zoneData` itself as a dependency — see the `hooks/` folder-structure entry above.

**Layout formula** — each `ZoneLayout` is computed, not stored, from two fixed per-`ZoneId` anchors (`ZONE_ANCHOR`: `centerX`, `centerY`; `ZONE_CELL_SIZE`: `cellWidth`, `cellHeight`, both private to `data/zones.ts`, derived from the original Figma numbers so the default grid renders pixel-identical to before this setting existed): `width = cellWidth * cols`, `height = cellHeight * rows`, `x = centerX - width / 2`, `y = centerY - height / 2`. So resizing grows/shrinks the zone symmetrically around its own fixed center on *both* axes — points never drift toward one side or one neighbor as the grid changes. (An earlier version anchored the top edge instead of `centerY`, growing only downward — that let a zone's box overlap the one below it as rows grew, or open a gap as rows shrank, e.g. Shoulder into Belly; centering the vertical growth the same way as the horizontal keeps a resize's effect on a neighbor roughly half of what pure top-anchoring would cause.)

**Shoulder-zone overlap avoidance.** Widening either the shoulder or belly zone's own columns (both grow symmetrically around a fixed `centerX`, per the formula above) can push the shoulder zone's box far enough right, or the belly zone's box far enough left, that the two start overlapping on screen — the shoulder zone's `centerY` was only ever tuned against `DEFAULT_ZONE_POINT_COUNTS`, so it carries no built-in margin for that. `buildZoneLayout` (`data/zones.ts`) raises both shoulder zones (moves them further up, i.e. lowers their `y`) whenever `shouldRaiseShoulderZones` is true: shoulder `cols` + belly `cols` > 4 (equivalent to, and simplifying, "shoulder `cols > 1` **and** belly `cols > 2`, **or** belly `cols > 3` on its own" — widening the shoulder zone itself only reaches the belly zone once the belly zone is already at least its default width; widening the belly zone alone past its default reaches the shoulder zone regardless of the shoulder zone's own width). When raised, `raisedShoulderCenterY` replaces the shoulder zone's fixed `centerY` with one computed to keep its box's bottom edge `SHOULDER_BELLY_MIN_GAP` above the belly zone's *current* top edge (which itself moves as the belly zone's row count changes) — both belly zones share one `centerY`/`cellHeight`, so either can be read regardless of which shoulder zone is being laid out. At `DEFAULT_ZONE_POINT_COUNTS` the condition is false and the formula above is unchanged.

**Belly-zone overlap avoidance.** The same symmetric-around-`centerX` growth also eventually closes the gap between the belly-right and belly-left zones themselves as belly `cols` grows past its default (3) — their fixed anchors only carry `BELLY_ZONES_MIN_GAP` of margin at that default width. `buildZoneLayout` widens both belly zones' *horizontal* placement whenever `shouldWidenBellyGap` is true (belly `cols > 3`): `widenedBellyCenterX` replaces a belly zone's fixed `centerX` with one placing its inner (toward-midline) edge exactly `BELLY_ZONES_MIN_GAP / 2` from `BELLY_ZONES_MIDLINE_X`, keeping the two zones symmetric around it the same way their fixed anchors already are. `BELLY_ZONES_MIDLINE_X` is the midpoint of the two belly zones' own fixed `centerX` anchors (0.5075) — deliberately *not* the generic `BODY_MIDLINE_X` (0.5, used only to classify which side of the body a zone falls on) — since the belly zones' actual illustrated center (relative to the navel in the body illustration) sits off of `BODY_MIDLINE_X` by design in the original Figma rectangles; widening around `BODY_MIDLINE_X` instead visibly drifted the gap off-navel. `BELLY_ZONES_MIN_GAP` (0.025) isn't an arbitrary new value either — it's the exact gap the two belly zones already have today at `DEFAULT_ZONE_POINT_COUNTS`'s cols (3), so nothing changes below that width. This is independent of, and combinable with, the shoulder-zone raise above — a config can trigger either, both, or neither.

**Shoulder+belly row-height compensation.** A tall combined shoulder+belly row count grows both zones' boxes noticeably taller than at `DEFAULT_ZONE_POINT_COUNTS`, since each grows symmetrically around its own fixed `centerY`. `buildZoneLayout` nudges *both* the shoulder and belly zones' `centerY` down by `SHOULDER_BELLY_ROWS_LOWER_OFFSET` (0.03) whenever `shouldLowerShoulderAndBellyZones` is true (shoulder `rows` + belly `rows` > 6) **and** the shoulder zones are already being raised (`shouldRaiseShoulderZones` above) — the row-height nudge only applies on top of an existing column-driven raise, not on its own. Shoulder rows can only shrink from their default/max (3, per `ZONE_MAX_GRID`), so in practice this only fires at belly `rows` = 4 with shoulder `rows` still at 3 (3 + 4 = 7) — the one grid shape whose combined row count exceeds the default sum (3 + 3 = 6) — and only once the current `cols` also satisfy the raise condition above. This offset is independent of, and combinable with, the belly-zone widen rule below (which is driven by `cols` alone).

**Point identity — and address — survives a resize.** Point ids are keyed by their `(row, column)` slot within the zone's grid (e.g. `br-r2c3`), not a flat sequential index — a flat index's row/col meaning changes whenever `cols` changes, so it can't be stable across a resize. Critically, this `column` *is* the body-relative address column (1 = closest to the body's own midline, `isZoneLeftOfMidline` in `data/zones.ts`) baked in at generation time, not a raw left-to-right position re-flipped against the *current* `cols` on every read — so a given point's own `PointContextMenu` row/place never changes just because some *other* column was added or removed elsewhere in the same row. Resizing only ever adds/removes the outermost (farthest-from-center) column or bottommost row; every nearer-to-center point keeps its exact id and address. `buildZoneData` derives left-to-right rendering order from this same column per zone side (descending for a zone left of the midline, since its closest-to-center column renders rightmost; ascending otherwise) — see its `rowColumns` computation.

Shrinking a zone's grid just stops rendering/using the now-out-of-range slots; their `StoredPointState` is left untouched in AsyncStorage (same "orphaned keys are harmless" philosophy `normalizeStorage` already relied on), so growing the grid back reveals that history again instead of resetting it. `setZonePointCounts` (`useAppStore.ts`) and `applyImport`'s `zonePointCounts` branch both re-run `normalizeStorage` against the new active-points list to backfill default states for newly-in-range slots only, leaving every other point's state as-is.

### Enabling/disabling individual zones

Each of the 6 zones can be individually shown or hidden via the **"Активные зоны"** row in the menu (`ZonesDialog`, opened from `SettingsSheet`), one collapsed-by-default `Accordion` per `ZoneType` (Плечи/Живот/Бёдра — the same three groups as `ZonePointsDialog`), each with a bulk header `Checkbox` (built the same way `AppDataSelector`'s "Настройки приложения" accordion is — a `Checkbox` passed directly as `Accordion`'s `label`, bulk-toggling both zones of that type and going indeterminate when only one is checked) plus two per-side `Checkbox` rows (e.g. Левое плечо/Правое плечо, `ZONE_TYPE_ZONE_IDS[type]` in `data/zones.ts`, right then left). It's backed by `EnabledZones` (`Record<ZoneId, boolean>`, `src/types/zone.ts`) and defaults to every zone enabled (`DEFAULT_ENABLED_ZONES` in `src/data/zones.ts`). At least one zone must always stay enabled: `ZonesDialog`'s `toggleZone`/`toggleType` reject (no-op) a toggle that would leave every zone off (`allDisabled` in its `utils/`), and `storage/utils/clampEnabledZones.ts` falls back to `DEFAULT_ENABLED_ZONES` entirely if a stored or imported value somehow has none set — a fully-disabled state would leave the body diagram with nowhere left to mark an injection. It's persisted like `zonePointCounts` (its own AsyncStorage key, round-trips through export/import as `ExportedAppData.enabledZones`/`ExportSettingKey.EnabledZones`) — see "Selective export / merge import / clear" above.

`SettingsSheet`'s own "Активные зоны" row mirrors "Точки в зонах" above, with one difference: once *every* zone is enabled (`SettingsSheet/utils/isAllZonesActive.ts`, all 6 `EnabledZones` entries `true`), the right-side value reads the fixed `menu.allZonesActiveValue` ("Все") instead of the plain count, and the below-row summary is omitted entirely (`buildActiveZonesSummary.ts` returns `[]`) — a "which sides are enabled" summary has nothing to add once the answer is "all of them". Short of that, it's a short right-side count as a plain number, no i18next key (`SettingsSheet/utils/countActiveZones.ts`, simply the number of `true` entries in `EnabledZones`) and, below the row, one line per `ZoneType` with at least one enabled side (`SettingsSheet/utils/buildActiveZonesSummary.ts`), e.g. "Плечи: левое, правое" / "Живот: слева" — the label (`ZONE_TYPE_LABEL_KEY`) and the comma-joined enabled sides (`ZONE_SIDE_LABEL_KEY[zoneId]` per enabled `ZoneId`, in `ZONE_TYPE_ZONE_IDS` order, i.e. right then left) interpolated into the existing `toast.labeledValue` template ("{{label}}: {{value}}") rather than a new key, since the shape is identical. A `ZoneType` with neither side enabled is left out entirely, and the fully-disabled case falls back to the shared `menu.allZonesDisabledSummary` sentence — see "Configurable point grid" above for the point-count row's identical fallback. This fully-disabled state is normally unreachable (`toggleZone`/`toggleType`'s `allDisabled` guard and `clampEnabledZones`'s fallback both prevent it), but the summary still handles it defensively rather than assuming it can't happen.

Disabling a zone gets the same treatment `ZonePointsDialog` gives a shrunk grid: `buildZoneData(zonePointCounts, enabledZones)` excludes a disabled zone's points from `points`/`pointMap`/`pointAddress`/`pointsByZone` entirely (so it's left out of `normalizeStorage`'s active-points list and out of the group checkmark indicators, `lastPressedByGroup`), while its `StoredPointState` is left untouched in AsyncStorage — re-enabling the zone reveals that history again, the same "orphaned keys are harmless" philosophy the point-grid setting already relies on. `zoneLayout` is still computed for a disabled zone unconditionally (`buildZoneLayout` has no notion of `enabledZones`); it's `MainScreen` that actually hides it, by filtering `ZONES` down to `state.enabledZones[zone.id]` before mapping to `ZoneContainer`, rather than relying on an empty/missing layout entry. Because `zoneData` now depends on two settings at once, `setZonePointCounts`/`setEnabledZones`/`applyImport`'s combined branch each re-run `normalizeStorage` against the *current* value of whichever setting isn't the one being changed (read from `prev`/`data ?? prev`), so a file that imports only one of the two still backfills correctly against the other's live value instead of a stale one.

Left/right zones of the same body part share one accent/glow colour pair, so the pair itself is keyed by `ZoneType` (`Shoulder`/`Belly`/`Thigh`, `src/types/zone.ts`) rather than duplicated per `ZoneId` — `ZONE_TYPE` (`Record<ZoneId, ZoneType>`, `src/data/zones.ts`) maps each of the 6 zones to its shared type. The colour pair itself lives on `ThemeColors.zoneColors` (`Record<ZoneType, ZoneColorPair>`, `src/theme/types.ts`), since it now varies by theme: `DARK_COLORS.zoneColors` is unchanged from the original Figma design; `LIGHT_COLORS.zoneColors` steps one shade darker per type (`accent`/`glow` share one value there — shoulder/thigh reuse `DARK_COLORS`' own `glow`, belly's green is a further manual step darker) since the original pair reads too faint against the light theme's bright background. `ZoneContainer`'s block (fill + border, both a `${accent}${alpha}` string) and `InjectionPoint`'s glow halo look up `colors.zoneColors[ZONE_TYPE[zoneId]]` via `useTheme()` (both only have a `ZoneId` prop, hence the `ZONE_TYPE` lookup); `HelpSheet`'s zone-legend badge/label text stores a `ZoneType` directly on each `INJECTION_ZONE_INFO` row and reads `colors.zoneColors[zone.type]`, skipping that lookup since it never has a specific `ZoneId` to begin with (left/right share one row). Either way `colors` already resolves to `DARK_COLORS`/`LIGHT_COLORS`, so no separate theme branching is needed for the colour itself. `ZoneContainer` additionally branches on `resolvedScheme` for its own fill/border alpha (`LIGHT_FILL_ALPHA`/`LIGHT_BORDER_ALPHA` vs `DARK_FILL_ALPHA`/`DARK_BORDER_ALPHA`, both in `ZoneContainer.tsx`), since its translucent wash needs a higher alpha in light theme on top of the darker colour; `HelpSheet`'s badge/text render at full opacity already, so no separate alpha is needed there.

### Point address

Each point has a body-relative address — `row` (1-indexed, top to bottom within its zone) and `column` (1-indexed, counted outward from the body's own vertical midline, 1 = closest to center) — shown in `PointContextMenu` (long-press menu, via an `address` prop computed by `MainScreen`) and in the toast shown after marking a point (tap-to-press, or the context menu's "Отметить" dialog — see `buildMarkToastMessage` in `MainScreen.tsx`, which also includes the zone label). Computed by `buildZoneData` in `src/data/zones.ts` directly from each point's generation-time `(row, column)` (the column is flipped to count from center depending on whether the zone's canonical, unmirrored position falls left or right of the body midline `x = 0.5`) — see "Configurable point grid" above. The address is fixed to the point's own anatomy and does **not** change when mirror mode is toggled — mirroring only changes which screen half a zone renders on, never the point's row/center-relative position.

The mark-confirmation toast (`buildMarkToastMessage`) adds two conditional lines on top of the address: it re-runs the pure `PointService.onPress` (with the caller's real `daysToWhite`/`daysToAvailable`/`pointRestoreMode` — see "Days-to-available setting" above) against the pre-mark point state to detect a `PressResultType.Blackout` outcome (site reused too early) and calls that out along with the triggered `blackoutDurationDays` (via `t('toast.markBlackoutSuffix', { count: days })`, i18next-pluralized — see "Localization" above), and it compares the marked timestamp against `Date.now()` — if it's more than `MARK_BACKDATED_THRESHOLD_MS` (`src/constants.ts`) in the past, the toast also shows the marked date/time via `formatDateTime` (`src/utils/formatDateTime.ts`, shared with `PointContextMenu`'s info lines). A plain tap is never backdated (its timestamp is always "now"); only the context menu's "Отметить" dialog (custom date/time picker) can trigger the backdated line — and, since that dialog lets the user pick any timestamp back to the point's `lastInjectionAt` with no re-check against the days-to-available window at that specific picked time (`PointContextMenu`'s own "Отметить" pre-check only gates on the point's availability *now*), `onPress`'s re-run here can come back `Blocked`/`Unavailable` even though the point was available when the dialog was opened — `buildMarkToastMessage` returns `null` (no toast) in that case rather than showing a misleading success/blackout message, matching `MainScreen`'s subsequent `actions.markPointAt` call, which silently no-ops on the same `Blocked`/`Unavailable` result.

### Group indicators

- **thighs** group: checkmark (✓) shown on the most recently used point across both thigh zones.
- **shoulders-and-belly** group: checkmark shown on the most recently used point across shoulders + belly zones.

Derived by scanning `events[]` from newest to oldest, skipping `manual-block` / `manual-unblock` events.
